# 方案设计文档 — android-stackblur HarmonyOS 移植

---

## 一、背景与目标

### 1.1 背景

android-stackblur 是由 Enrique López Mañas 封装的高性能图像模糊处理库，实现了 Mario Klingemann 的 Stack Blur 算法。该算法在 Android 应用中被广泛用于背景虚化、毛玻璃效果等 UI 场景。为支持鸿蒙应用开发者使用同等能力，需将其适配到 HarmonyOS API 12+。

### 1.2 移植目标

- **功能目标**：提供与 Android 侧语义等价的 Stack Blur API，支持 ArkTS 实现（CPU 路径）和 NAPI C 实现（Native 路径）两种处理方式
- **兼容目标**：公开接口名称与 Android 侧保持一致（`StackBlurManager`、`process()`、`processNatively()`），仅做图像类型和异步模型的必要适配
- **质量目标**：全部测试用例 PASS；Native 路径正确性与 ArkTS 路径一致（像素差 < 1%）
- **交付物**：HAR 包（`library` 模块）+ Demo 示例页（`entry/src/main/ets/pages/Index.ets`）+ ohosTest 测试用例

### 1.3 范围

| 纳入范围 | 排除范围 |
|----------|----------|
| Stack Blur 核心算法（ArkTS 实现） | RenderScript GPU 路径（`RSBlurProcess`）— HarmonyOS 不支持 |
| C 层 `stackblurJob()` 算法复用 + NAPI 封装（Native 路径） | 多线程并行模糊（Android 的 ExecutorService 多核方案） |
| `StackBlurManager` 公开类（精简为 5 个方法） | `saveIntoFile()` — 由调用方使用 OH 文件 API 实现 |
| Demo 示例和单元测试 | |

## 二、Android 库分析总结

### 2.1 库整体架构

```
调用方
  └──► StackBlurManager(Bitmap)
            │
            ├─ process(radius)          ──► JavaBlurProcess.blur()
            │                                  └─ EXECUTOR.invokeAll(horizontal BlurTask[])
            │                                  └─ EXECUTOR.invokeAll(vertical BlurTask[])
            │                                       └─ blurIteration() [stackblur_mul/shr 查找表]
            │
            ├─ processNatively(radius)  ──► NativeBlurProcess.blur()
            │                                  └─ EXECUTOR.invokeAll(horizontal NativeTask[])
            │                                  └─ EXECUTOR.invokeAll(vertical NativeTask[])
            │                                       └─ JNI: functionToBlur(Bitmap, ...) → stackblurJob() [blur.c]
            │
            └─ processRenderScript(ctx, radius) ──► RSBlurProcess.blur()
                                                       └─ RenderScript API + ScriptC_blur (blur.rs)
                                                       └─（失败时回退 NativeBlurProcess）
```

关键设计：  
- 所有路径均为两次独立 pass（水平 + 垂直），可多线程并行  
- C 层 `stackblurJob()` 按 `step` 参数区分水平（step=1）或垂直（step=2） pass  
- 查找表 `stackblur_mul[255]` / `stackblur_shr[255]` 同时存在于 Java 层和 C 层（相同内容，避免 JNI 数据传输）

### 2.2 可移植性分析结论

| 分析维度 | 结论 | 说明 |
|----------|------|------|
| **复杂度评级** | MEDIUM | C 核心算法完全可复用；主要工作是 JNI→NAPI 和 Bitmap→PixelMap 适配 |
| **Android 专有 API 依赖** | 有（轻度） | `android.graphics.Bitmap`（图像数据容器）、`android.content.Context`（仅 RS 路径使用，用于排除范围） |
| **JNI/NDK 依赖** | 有 | `<jni.h>`、`<android/bitmap.h>`、`AndroidBitmap_lockPixels()` — NAPI 迁移 |
| **UI 组件依赖** | 无 | 库本身不含 UI，Demo 使用 Android View（移植 Demo 需重写） |
| **外部三方库依赖** | 无（库模块） | Demo 依赖 RoboGuice，库模块无外部依赖 |
| **移植可行性** | 完全可移植 | C 核心算法 100% 复用，JNI 层替换为 NAPI，Bitmap 替换为 PixelMap |

### 2.3 关键迁移挑战

| # | 挑战描述 | 复杂度 | 解决方向 |
|---|----------|--------|---------|
| 1 | `Bitmap` 像素数据获取与 `PixelMap` 像素数据获取方式不同；HarmonyOS C 层通过 `OH_PixelmapNative_ReadPixels()` / `OH_PixelmapNative_WritePixels()` API 访问像素 | HIGH | NAPI 层使用 `@kit.ImageKit` 的 NDK API 操作 PixelMap 像素数据；ArkTS 层通过 `pixelMap.readPixelsToBuffer()` / `pixelmap.writePixels()` 操作 |
| 2 | `AndroidBitmap_lockPixels()` 无 HarmonyOS 等价接口 | HIGH | 改为通过 NAPI 接收 ArrayBuffer（像素数据），C 层直接操作 ArrayBuffer 内存，处理完成后通过 NAPI 回传；避免在 C 层直接持有 PixelMap 指针 |
| 3 | Android C 层引用了 `<android/log.h>` 和 `<android/bitmap.h>` | MEDIUM | 替换为 `<hilog/log.h>` 日志宏（使用 `%{public}` 修饰符），移除 `<android/bitmap.h>` 和相关 JNI 绑定代码 |
| 4 | 异步模型：`process()` Android 侧为同步，OH 侧主线程不能阻塞 | MEDIUM | 返回 `Promise<image.PixelMap>`，内部通过 `taskpool.execute()` 在后台线程执行算法 |

## 三、整体移植方案

### 3.1 移植路径选择

- [x] **路径 B：平台 API 替换** — ArkTS 翻译 Java 路径，替换 `Bitmap` 操作为 `PixelMap` 操作
- [x] **路径 C：Native 代码复用 + NAPI 封装** — C 核心算法直接复用，替换 JNI 绑定层为 NAPI

**放弃路径 D（UI 重写）**：库本身无 UI。  
**放弃 RenderScript 路径**：已废弃，且 HarmonyOS 无对应 API；Native C 路径性能足够替代。

### 3.2 整体架构设计

```
调用方（ArkTS 应用）
  └──► StackBlurManager(pixelMap: image.PixelMap)   [library/src/main/ets/StackBlurManager.ets]
            │
            ├─ process(radius): Promise<PixelMap>
            │     taskpool.execute(...)
            │       └─ pixelMap.readPixelsToBuffer() → Uint8Array
            │       └─ stackBlurJobArkTS(buf, w, h, radius, ...) [单线程 ArkTS 实现]
            │       └─ image.createPixelMap(buf, opts) → 新 PixelMap
            │
            └─ processNatively(radius): Promise<PixelMap>
                  taskpool.execute(...)
                    └─ pixelMap.readPixelsToBuffer() → ArrayBuffer
                    └─ nativeBlur(buf, w, h, radius)  [NAPI → libblur.so → blur_core.c]
                    └─ image.createPixelMap(buf, opts) → 新 PixelMap
```

## 四、核心模块移植方案

### 4.1 模块：blur_core.c — C 层核心算法

**Android 实现**：`StackBlur/src/main/cpp/blur.c`，包含 `stackblurJob()` 和 JNI 绑定函数。

**HarmonyOS 方案**：
- **直接复用**（不修改逻辑）：`stackblurJob()` 函数体及 `stackblur_mul[255]`、`stackblur_shr[255]` 查找表
- **删除**：整个 JNI 绑定函数 `Java_com_enrique_stackblur_NativeBlurProcess_functionToBlur()`
- **替换头文件**：
  - 删除 `#include <jni.h>`、`#include <android/bitmap.h>`、`#include <android/log.h>`
  - 添加 `#include <hilog/log.h>`
  - 删除 `ANDROID_LOG_INFO/ERROR` 宏，添加 OH 日志宏（使用 `%{public}` 修饰符）
- **删除** `#define LOG_TAG`、`LOGI`、`LOGE` 宏（在 napi_blur.cpp 中重新定义）
- **新增文件**：`napi_blur.cpp` — NAPI 绑定层，接收 ArrayBuffer + w + h + radius 参数

**关键 API 替换**：

| Android C 接口 | HarmonyOS C 接口 | 说明 |
|---------------|-----------------|------|
| `#include <android/log.h>` | `#include <hilog/log.h>` | 日志头文件 |
| `__android_log_print(ANDROID_LOG_INFO, tag, fmt, ...)` | `OH_LOG_INFO(LOG_APP, fmt, ...)` | 日志输出（需 `%{public}` 修饰符）|
| `#include <android/bitmap.h>` | 删除（像素数据通过 ArrayBuffer 传入） | 无对应头文件 |
| `AndroidBitmap_lockPixels()` | 删除（ArrayBuffer 直接操作） | NAPI 层直接读取 ArrayBuffer 指针 |
| `AndroidBitmap_unlockPixels()` | 删除 | 同上 |

---

### 4.2 模块：napi_blur.cpp — NAPI 绑定层

**Android 实现**：`blur.c` 中的 `Java_com_enrique_stackblur_NativeBlurProcess_functionToBlur()` 函数（接收 `jobject Bitmap`、`jint radius`、`jint threadCount`、`jint threadIndex`、`jint round`，通过 `AndroidBitmap_lockPixels()` 获取像素指针后调用 `stackblurJob()`）。

**HarmonyOS 方案**：新建 `napi_blur.cpp`，注册导出函数 `nativeBlur`。

**NAPI 函数签名设计**：
```
nativeBlur(buffer: ArrayBuffer, width: number, height: number, radius: number): void
```
- `buffer`：RGBA_8888 格式像素数据的 ArrayBuffer（total size = w * h * 4）
- `width`、`height`：图像尺寸
- `radius`：模糊半径（1–254）
- 函数 in-place 修改 buffer 内容，无返回值

**NAPI 实现关键步骤**：
```cpp
// 1. 解析 4 个参数
napi_get_cb_info(env, info, &argc, args, nullptr, nullptr);
// 2. 获取 ArrayBuffer 数据指针和字节长度
void* data = nullptr;
size_t byteLen = 0;
napi_get_arraybuffer_info(env, args[0], &data, &byteLen);
// 3. 解析 width、height、radius
int32_t w, h, radius;
napi_get_value_int32(env, args[1], &w);
napi_get_value_int32(env, args[2], &h);
napi_get_value_int32(env, args[3], &radius);
// 4. 直接调用 stackblurJob（单线程，水平 + 垂直两次 pass）
stackblurJob((unsigned char*)data, w, h, radius, 1, 0, 1);
stackblurJob((unsigned char*)data, w, h, radius, 1, 0, 2);
```

**CMakeLists.txt 关键配置**：
```cmake
cmake_minimum_required(VERSION 3.5.0)
project(stackblur)

add_library(blur SHARED         # 生成 libblur.so
    napi_blur.cpp
    blur_core.c
)

target_link_libraries(blur PUBLIC
    libace_napi.z.so
    libhilog_ndk.z.so
)
```

**index.d.ts 类型声明**：
```typescript
export const nativeBlur: (buffer: ArrayBuffer, width: number, height: number, radius: number) => void;
```

### 4.3 模块：StackBlurManager.ets — ArkTS 主类

**Android 实现**：`StackBlurManager.java`（持有 Bitmap、调用三种 BlurProcess）+ `JavaBlurProcess.java`（多线程 Stack Blur ArkTS 翻译目标）。

**HarmonyOS 方案**：单文件 `StackBlurManager.ets`，包含：
1. 模块级常量 `STACKBLUR_MUL` / `STACKBLUR_SHR`（与 Java/C 查找表相同内容）
2. 内部函数 `stackBlurJobArkTS()`（将 Java `blurIteration()` 逐行翻译为 ArkTS）
3. `StackBlurManager` 导出类

**`process()` 方法实现（ArkTS 路径）**：
```typescript
async process(radius: number): Promise<image.PixelMap> {
  if (radius < 1 || radius > 254) {
    return Promise.reject(new Error(`INVALID_RADIUS: ${radius} must be in [1, 254]`));
  }
  const info = this._pixelMap.getImageInfoSync();
  const w = info.size.width;
  const h = info.size.height;
  const buf = new ArrayBuffer(w * h * 4);
  await this._pixelMap.readPixelsToBuffer(buf);
  const src = new Uint8Array(buf);
  stackBlurJobArkTS(src, w, h, radius, 1, 0, 1); // 水平 pass
  stackBlurJobArkTS(src, w, h, radius, 1, 0, 2); // 垂直 pass
  const blurred = await image.createPixelMap(buf, { size: { width: w, height: h } });
  this._result = blurred;
  return blurred;
}
```

**`processNatively()` 方法实现（NAPI 路径）**：
```typescript
async processNatively(radius: number): Promise<image.PixelMap> {
  if (radius < 1 || radius > 254) {
    return Promise.reject(new Error(`INVALID_RADIUS: ${radius} must be in [1, 254]`));
  }
  const info = this._pixelMap.getImageInfoSync();
  const w = info.size.width;
  const h = info.size.height;
  const buf = new ArrayBuffer(w * h * 4);
  await this._pixelMap.readPixelsToBuffer(buf);
  nativeBlur(buf, w, h, radius);  // NAPI 调用，in-place 修改 buf
  const blurred = await image.createPixelMap(buf, { size: { width: w, height: h } });
  this._result = blurred;
  return blurred;
}
```

**关键 API 替换**：

| Android API | HarmonyOS API | 说明 |
|-------------|---------------|------|
| `original.getWidth()` / `getHeight()` | `pixelMap.getImageInfoSync().size.width/height` | 获取图像尺寸 |
| `original.getPixels(int[], 0, w, 0, 0, w, h)` | `pixelMap.readPixelsToBuffer(ArrayBuffer)` | 读取全部像素数据 |
| `Bitmap.createBitmap(pixels, w, h, Config.ARGB_8888)` | `image.createPixelMap(ArrayBuffer, {size:{width,height}})` | 从像素数据创建图像对象 |
| `original.copy(Config.ARGB_8888, true)` | `pixelMap.readPixelsToBuffer(buf)` + 操作 buf + `image.createPixelMap(buf, ...)` | 创建可写副本 |
| `int[]` 像素数组（ARGB 32 位 int） | `Uint8Array` / `ArrayBuffer`（RGBA 8 位/通道） | 每像素 4 字节，channel 顺序：R G B A |

**注意**：Android `JavaBlurProcess` 使用 `int[]` 存储像素（每个 int 为 ARGB），C 层使用 `unsigned char*`（每像素 4 字节 RGBA）。ArkTS 版本统一使用 `Uint8Array`（RGBA，与 C 层一致）。ArkTS 翻译时需将 Java 的位移操作（`>>> 16 & 0xff` 取 R 通道）改为 `src[i*4]` 直接索引。

## 五、关键技术决策

### 决策 1：NAPI 函数接收 ArrayBuffer 而非直接操作 PixelMap

**背景**：需要在 C 层对 PixelMap 像素数据执行 Stack Blur 算法。

**方案对比**：

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 方案 A：在 NAPI 层通过 `OH_PixelmapNative_ReadPixels()` 直接读取 PixelMap 像素 | 接口更简洁，ArkTS 侧不需要手动转 ArrayBuffer | `OH_PixelmapNative_ReadPixels()` API 在 HAR 中使用需链接 `libpixelmap_ndk.z.so`，且 API 版本兼容性需额外验证 | 放弃 |
| 方案 B：ArkTS 侧通过 `readPixelsToBuffer()` 读出 ArrayBuffer，传入 NAPI | 数据准备在 ArkTS 层完成，NAPI 只处理纯内存指针，接口稳定、测试方便 | 增加一次 ArkTS↔NAPI 数据传递 | 采用 |

**决策结论**：采用方案 B，在 ArkTS 层将 PixelMap 像素数据读出为 ArrayBuffer，传入 NAPI 的 `nativeBlur()` 函数进行 in-place 修改，然后用修改后的 ArrayBuffer 创建新的 PixelMap。

---

### 决策 2：`process()` 改为异步 Promise API

**背景**：Android 端 `process()` 为同步调用，HarmonyOS 主线程不能执行耗时计算。

**方案对比**：

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 保持同步，要求调用方自行使用 taskpool | 接口与 Android 侧完全一致 | 容易导致主线程卡顿，违背 OH 设计规范 | 放弃 |
| 在库内返回 Promise，调用方用 await | 安全，与 OH 异步生态一致；调用方代码简单 | 与 Android SDK 接口不兼容 | 采用 |

**决策结论**：`process()` 和 `processNatively()` 均返回 `Promise<image.PixelMap>`。

---

### 决策 3：ArkTS 查找表使用模块级常量（非类静态属性）

**背景**：`stackblur_mul[255]` 和 `stackblur_shr[255]` 是只读固定数组。

**决策结论**：声明为 `ets` 文件模块顶层 `const` 数组，初始化后不再修改，避免 class static 成员访问开销。

---

### 决策 4：不移植多线程并行（ExecutorService）路径

**背景**：Android 使用 `EXECUTOR.invokeAll()` 将水平/垂直 pass 分片到多核 CPU 上并行执行。

**方案对比**：

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 使用 OH `taskpool` 多 task 并行 | 极端大图下可提速 | 多任务需共享 ArrayBuffer，OH ArkTS 中 SharedArrayBuffer 在 taskpool 间共享有限制；复杂度大幅增加 | 放弃（第一版） |
| 单线程执行（当前方案） | 实现简单，正确性高，测试容易 | 多核利用率低 | 采用 |

**决策结论**：第一版使用单线程，若性能测试不达标再考虑 taskpool 多 task 方案。

## 六、API 差异与兼容性说明

### 6.1 破坏性变更（Breaking Changes）

| # | 变更描述 | Android 侧 | HarmonyOS 侧 | 迁移方法 |
|---|----------|-----------|-------------|---------|
| 1 | 构造参数类型 | `Bitmap` | `image.PixelMap` | 调用方从 `BitmapFactory.decodeResource()` 改为 `@ohos.multimedia.image` 的 `createImageSource().createPixelMap()` |
| 2 | `process()` 返回值和异步模型 | `Bitmap`（同步） | `Promise<image.PixelMap>` | 调用方改为 `const result = await manager.process(radius)` |
| 3 | `processNatively()` 返回值和异步模型 | `Bitmap`（同步） | `Promise<image.PixelMap>` | 同上 |
| 4 | `processRenderScript()` 不存在 | 存在，GPU 加速 | 不提供此方法 | 改用 `processNatively()`，性能无显著差距 |
| 5 | `saveIntoFile()` 不存在 | 存在 | 不提供此方法 | 调用方使用 `@ohos.file.fs` + `PixelMap.packing()` |

### 6.2 行为差异（非破坏性）

| # | 差异描述 | Android 侧行为 | HarmonyOS 侧行为 |
|---|----------|--------------|----------------|
| 1 | 半径校验 | 无校验，超出 `stackblur_mul` 数组长度（255）时行为未定义 | `radius < 1` 或 `radius > 254` 时 Promise reject (`INVALID_RADIUS`) |
| 2 | 处理模式 | 多线程（CPU 核数并行） | 单线程 |
| 3 | `returnBlurredImage()` 未调用时 | 返回 `null` | 返回 `null`（行为一致） |

## 八、风险评估

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| `image.createPixelMap(ArrayBuffer, opts)` 对坐标格式有额外要求（如需指定 pixelFormat） | 中 | 高 | 编译测试阶段验证，确认 `pixelFormat: image.PixelMapFormat.RGBA_8888` 参数是否必填 |
| C 层 `unsigned char*` 与 ArkTS `Uint8Array` 的 RGBA 字节序是否与 PixelMap 一致 | 中 | 高 | TC-011 一致性测试验证；如不一致需在 ArkTS 或 C 层加入字节序转换 |
| ArkTS 整数算术与 Java `int` / `long` 大数计算精度差异 | 低 | 中 | `sum_r/g/b` 最大值约 `254 * (254+1) * (254+1)^2 ≈ 4.2×10^9`，超出 `number` 安全整数范围；需使用 `BigInt` 或重构为不超出 `Number.MAX_SAFE_INTEGER` 的计算（NAPI C 路径不受影响） |
